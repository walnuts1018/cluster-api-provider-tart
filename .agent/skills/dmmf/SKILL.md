# Skill: Functional Domain Modeling (関数型ドメインモデリング)

## 概要 (Overview)

このSkillは、関数型プログラミングのパラダイムを用いてドメイン駆動設計（DDD）を実践するためのアーキテクチャ設計および実装ガイドラインです。
「不正な状態を型で防ぐ」「ビジネスプロセスをデータの変換パイプラインとして捉える」「副作用を分離する」という中核原則に従い、予測可能でテストしやすく、ドメインの知識がコード自体に表現された堅牢なシステムを構築することを目的とします。

## コア原則 (Core Principles)

1. **ユビキタス言語の型へのマッピング (Ubiquitous Language into Types)**
   - ドメインエキスパートが使う言葉を、そのままコード上の「型」として表現する。
2. **不正な状態を表現不能にする (Make Illegal States Unrepresentable)**
   - 実行時のバリデーションに頼るのではなく、コンパイル時点で不正な状態が存在し得ないようにデータ構造（型）を設計する。
3. **ビジネスルールを関数としてモデル化する (Business Rules as Functions)**
   - ビジネス上のワークフロー（手続き）を、「入力から出力への純粋なデータ変換（関数）」として表現する。
4. **副作用の分離 (Separation of Side Effects)**
   - ドメインロジック（純粋）と、I/Oや状態変更などの副作用（非純粋）を明確に分離し、副作用をシステムの境界（外側）に押しやる。
5. **鉄道指向プログラミング (Railway Oriented Programming)**
   - エラーハンドリングを制御フロー（例外）として扱わず、データフロー（戻り値）としてモデル化する。

## 設計・実装ガイドライン (Design & Implementation Guidelines)

### 1. 型によるドメインのモデル化 (Modeling with Types)

基本的なデータ型（String, Intなど）への執着（Primitive Obsession）を避け、ドメインの意味を持つ型を定義してください。

- **直積型 (Product Types / Records / Structs):** 複数のデータが「同時に（AND）」存在する必要がある場合に使用します。Goでは`struct`に相当します。
  - _例:_ `User = Name AND EmailAddress`
- **直和型 (Sum Types / Enums):** 複数の状態のうち「いずれか1つだけ（OR）」が存在する場合に使用します。Goでは`interface` + concrete type、または`const` enumで表現し、`switch`文で網羅性チェックを行います。
  - _例:_ `PaymentMethod = CreditCard OR BankTransfer OR Cash`
- **制約付きの型 (Constrained Types / Value Objects):**
  - 単なる文字列や数値ではなく、生成時にビジネスルール（文字数、フォーマットなど）を満たしていることのみを保証するラッパー型を作成します。一度生成されれば、その型を受け取る関数は再度のバリデーションを必要としません。

### 2. 状態遷移のモデリング (State Machines via Types)

状態の変更をオブジェクト内部のフラグ（`isPaid = true`, `status = "shipped"`）を更新することで表現しないでください。代わりに、**状態ごとに異なる型**を定義し、状態遷移を「古い型のオブジェクトを受け取り、新しい型のオブジェクトを返す関数」として表現します。

- _アンチパターン:_ `Order` クラスの `isValidated` フラグを true にする。
- _推奨パターン:_ `UnvalidatedOrder` を受け取り、成功すれば `ValidatedOrder` を返す関数を定義する。これにより、支払い処理の関数は `ValidatedOrder` 型しか受け付けないようにコンパイラレベルで強制できます。

### 3. パイプラインとしてのワークフロー (Workflow as a Pipeline)

ビジネスユースケースを小さな純粋関数の組み合わせ（合成）として構築します。
データは一方向に流れ、変更されるのではなく常に新しく生成（イミュータブル）されます。

- **入力 (Command) -> 検証 -> ビジネスロジック -> 出力 (Domain Event または Result)**

### 4. 予測可能なエラーハンドリング (Predictable Error Handling)

ビジネス上のエラー（在庫不足、残高不足など）を表現するために「例外 (Exceptions)」を投げないでください。例外はシステムの中断（パニック）やインフラエラーのために予約します。
代わりに、成功と失敗の両方を表現できる型を明示的に返してください。Goでは`(..., error)`の形式を用い、エラー型をドメイン固有の型として定義することで、ビジネスエラーとシステムエラーを区別してください。

### 5. 副作用の境界づけとI/O (Isolating I/O)

ドメインモデル（中心核）を完全に純粋（Pure）に保ちます。

- ドメイン関数の中で、データベースへのクエリ発行、外部API呼び出し、システム時刻の取得、ランダム値の生成を行ってはいけません。
- 必要なデータ（現在のDBの状態など）は、ドメイン関数を呼び出す**前**にインフラ層で取得して引数として渡し、ドメイン関数は「何を保存すべきか（イベントやデータ）」を戻り値として返します。その戻り値を使って、関数の**後**でインフラ層がI/Oを実行します。

## エージェント向け振る舞い指示 (Instructions for the AI Agent)

あなたがアーキテクチャの設計、コードの生成、リファクタリング、またはコードレビューを行う際は、以下のルールに厳密に従ってください：

1.  **Think in Types First:**
    コードのロジック（実装）を書き始める前に、必ず要件から「ドメインの型（直積型、直和型）」を抽出し、型の定義から設計を始めてください。
2.  **No Mutation:**
    変数の再代入やオブジェクトのミューテーション（破壊的変更）を避け、可能な限りイミュータブル（不変）なデータ構造を使用してください。
3.  **Enforce Exhaustiveness:**
    状態の分岐を扱う際、デフォルトフォールバック（`else` や `default`）を安易に使用せず、`switch`文で既知の各状態を明示的に列挙してください。Goでは`interface` + concrete typeで直和型を表現し、`switch`の各caseで状態を明示的に処理します。未知の値が到達する場合（本来到達不能）は、エラーとして返却してください。
4.  **Make Illegal States Unrepresentable:**
    ユーザーからのプロンプトで「Aの場合はBフィールドは空になる」といった仕様が提示された場合、オプショナルフィールドで表現するのではなく、Aの状態とそれ以外の状態を別々の型（直和型）に分離する提案を行ってください。
5.  **Pure Domain Logic:**
    生成するビジネスロジックの関数内に依存関係（DBリポジトリやHTTPクライアント）を直接注入しないでください。副作用のある操作と純粋なビジネスロジックを分離する設計（例: 関数型コア・命令型シェル）を提示してください。
6.  **Explicit Failure Modes:**
    ビジネスルール違反を表現するために例外を使用せず、戻り値の型シグネチャ（`(..., error)`）で失敗の可能性を明示してください。エラー型はドメイン固有の型として定義し、ビジネスルール違反とシステムエラーを区別してください。
